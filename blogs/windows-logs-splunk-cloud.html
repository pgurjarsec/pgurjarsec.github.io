Introduction

As a Cyber Security student learning SOC operations, one of the first hands-on skills you need is getting data into a SIEM. Splunk Cloud is one of the most widely used SIEM platforms in the industry, and Windows Event Logs are the most common data source in enterprise environments.

In this guide, I'll walk you through the complete process of sending Windows Event Logs (Application, Security, and System) to your Splunk Cloud Trial account — from creating indexes to verifying data flow. I documented every step while doing this myself, including the real errors I hit and how I fixed them.

// What you will achieve
Your Windows laptop will send live Event Logs to Splunk Cloud in real time. You will be able to search and analyze them using SPL (Splunk Processing Language).

Prerequisites

  • Windows 10 or 11 laptop with Administrator access
  • A free Splunk Cloud Trial account — sign up here
  • Notepad++ installed — download here (do NOT use regular Notepad)
  • Port TCP 9997 outbound open on your network/firewall
  • PowerShell running as Administrator
⚠ Important
Always run PowerShell as Administrator throughout this guide. Right-click PowerShell → "Run as administrator".
STEP 01 Log in to Splunk Cloud Trial

Open your browser and navigate to your Splunk Cloud trial URL:

https://prd-p-XXXXXX.splunkcloud.com

Log in with your admin credentials. You should land on the Splunk Cloud home dashboard.

STEP 02 Create Index in Splunk Cloud

Splunk stores data in indexes. You need to create one before data can be sent to it. Think of an index like a dedicated folder for a specific type of log.

  1. Click Settings in the top navigation
  2. Click Indexes
  3. Click New Index

Fill in the following fields:

FieldValue
Index Namewineventlog
Index Data TypeEvents
Searchable Time (days)30

Click Save.

// Important
The index name wineventlog must match exactly what you put in inputs.conf later — same spelling, all lowercase.
STEP 03 Download Universal Forwarder Credentials

The Universal Forwarder Credentials file (splunkclouduf.spl) contains your Splunk Cloud instance address, port 9997 endpoint, and SSL certificates — all pre-configured. This tells your forwarder where to send data.

  1. Click Apps in the top navigation
  2. Click Universal Forwarder
  3. Click "Download Universal Forwarder Credentials"

Save the downloaded file. It will be named splunkclouduf.spl.

STEP 04 Download & Install Universal Forwarder

Download

Go to the Splunk Universal Forwarder download page and select Windows 64-bit .msi:

https://www.splunk.com/en_us/download/universal-forwarder.html

Install

  1. Double-click the downloaded .msi file
  2. Accept the License Agreement
  3. Select the "Splunk Cloud" option
  1. Set credentials:
FieldValue
Usernameadmin
PasswordSomething memorable — avoid @#$ special characters to prevent PowerShell issues
⚠ Remember This Password
You will need this password when running CLI commands later. If it contains special characters like @ or #, wrap it in single quotes in PowerShell: 'MyPass@123'.
  1. Leave Deployment Server blank
  2. Click Next → Install
STEP 05 Extract & Install Forwarder Credentials

The splunkclouduf.spl file is a .tar.gz archive. We extract it and copy the contents into the forwarder's apps folder. Open PowerShell as Administrator and run each command one at a time:

1. Extract the credentials file

// PowerShell
tar -xzf "C:\Users\Dell\Downloads\splunkclouduf.spl" -C "C:\Users\Dell\Downloads\splunkclouduf_extracted"

2. Verify what was extracted

// PowerShell
ls "C:\Users\Dell\Downloads\splunkclouduf_extracted"

You should see a folder like 100_prd-p-kpfrw_splunkcloud.

3. Copy credentials to forwarder apps folder

// PowerShell
Copy-Item "C:\Users\Dell\Downloads\splunkclouduf_extracted\*" "C:\Program Files\SplunkUniversalForwarder\etc\apps\" -Recurse -Force

4. Restart the forwarder

// PowerShell
Restart-Service SplunkForwarder

5. Verify connection to Splunk Cloud

// PowerShell
cd "C:\Program Files\SplunkUniversalForwarder\bin"
.\splunk.exe list forward-server

Expected output:

Active forwards:
        inputs.prd-p-kpfrw.splunkcloud.com:9997 (ssl)
Configured but inactive forwards:
        None
// Success Signal
Seeing inputs.prd-p-kpfrw.splunkcloud.com:9997 (ssl) under Active forwards confirms your forwarder is connected to Splunk Cloud. This is a key milestone!
⚠ Login Failing at CLI?
If login keeps failing, it is often because of special characters in the password. Keep trying — it usually succeeds after 1–2 attempts. Or wrap the password in single quotes: -auth 'splunk:YourPass@123'.
STEP 06 Install Splunk Add-on for Windows

The Splunk Add-on for Windows provides the source types and field extractions needed to properly parse Windows Event Logs. Download it from Splunkbase and install it on your Splunk Cloud instance.

// Note
On Splunk Cloud trial, the add-on may already be available in the app inventory. Go to Apps → Find More Apps and search for "Splunk Add-on for Microsoft Windows" — if it's already installed you can skip the manual upload.
STEP 07 Create inputs.conf

The inputs.conf file tells the Universal Forwarder what data to collect. We configure it to collect Windows Event Logs from Application, Security, and System channels.

1. Create the app folder structure

// PowerShell
mkdir "C:\Program Files\SplunkUniversalForwarder\etc\apps\my_windows_inputs\local"

2. Open Notepad++ and create inputs.conf

Open Notepad++ → File → New → paste this content exactly:

// inputs.conf
[WinEventLog://Application]
disabled = 0
start_from = oldest
index = wineventlog
renderXml = true

[WinEventLog://Security]
disabled = 0
start_from = oldest
index = wineventlog
renderXml = true

[WinEventLog://System]
disabled = 0
start_from = oldest
index = wineventlog
renderXml = true

3. Save the file correctly

  1. Click File → Save As
  2. Navigate to: C:\Program Files\SplunkUniversalForwarder\etc\apps\my_windows_inputs\local\
  3. Set filename: inputs.conf
  4. Set Save as type: All Files (*.*)
  5. Click Save
⚠ Critical — File Extension
If you use regular Notepad or select "Text Documents", the file saves as inputs.conf.txt which Splunk cannot read. Always use Notepad++ and select All Files (*.*).

4. Verify the file

// PowerShell
ls "C:\Program Files\SplunkUniversalForwarder\etc\apps\my_windows_inputs\local\"

Should show inputs.conf — NOT inputs.conf.txt.

STEP 08 Restart Forwarder & Verify Logs in Splunk Cloud

1. Restart the forwarder

// PowerShell
Restart-Service SplunkForwarder

2. Wait 2–3 minutes

Give the forwarder time to start, read the new inputs.conf, and begin shipping events to Splunk Cloud.

3. Search in Splunk Cloud

  1. Log in to Splunk Cloud
  2. Go to Apps → Search & Reporting
  3. Enter this search:
// SPL Search
index=wineventlog earliest=-15m
  1. Set time range to Last 15 minutes
  2. Click the Search button
// You should see this
126 events appearing from your Windows machine — XML-formatted event data from Application, Security, and System logs. Congratulations, your logs are flowing! 🎉

Troubleshooting

Here are the common issues I encountered and how I fixed them:

ProblemCauseFix
Login failed in PowerShell CLI Special characters in password Wrap password in single quotes: 'MyPass@123' — or retry, it works after 1–2 attempts
inputs.conf not found Saved as inputs.conf.txt Run: Rename-Item inputs.conf.txt inputs.conf
No logs in Splunk Cloud Index name mismatch Verify index name is exactly wineventlog in both Splunk Cloud and inputs.conf
No active forwards Credentials folder missing Verify 100_prd-p-kpfrw_splunkcloud exists in etc\apps\
Cannot delete files during uninstall Service still running Run Stop-Service SplunkForwarder first, then use Control Panel to uninstall
Multiple commands fail at once Pasting all on one line Run each PowerShell command on a separate line, one at a time

Conclusion

You have successfully set up the Splunk Universal Forwarder on your Windows machine and configured it to send Windows Event Logs — Application, Security, and System — to your Splunk Cloud instance in real time.

This is a foundational SOC skill. In a real enterprise environment, you would deploy this across hundreds of machines using a Deployment Server, but the core concepts are exactly the same as what you did here.
